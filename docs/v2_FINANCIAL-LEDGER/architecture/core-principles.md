# 핵심 설계 원칙

## 1. 서비스가 아닌 프레임워크

### 철학
이 프로젝트는 "서비스"가 아닌 "프레임워크"입니다.

**서비스가 아님:**
- ❌ SaaS 형태 제공하지 않음
- ❌ 중앙 서버 운영하지 않음
- ❌ 사용자 데이터 수집하지 않음
- ❌ 구독료나 유료 기능 없음

**프레임워크임:**
- ✅ 사용자가 직접 배포
- ✅ 사용자가 직접 설정
- ✅ 완전한 제어권
- ✅ 개인정보 보호

---

## 2. Core / Module / Plugin 분리

### 계층 구조

```
Core (Core Layer)
 ├─ 절대 최소 변경
 ├─ 안정성 최우선
 └─ 하위 호환성 유지
    ↓
Modules (Feature Layer)
 ├─ 실제 기능 구현
 ├─ 독립적으로 개발
 └─ 언제든 추가/제거
    ↓
Plugins (Extension Layer)
 ├─ 실험적 기능
 ├─ 깨져도 무방
 └─ 선택적 사용
```

### 원칙

**Core는:**
- 자주 변경하지 않음
- 변경 시 충분한 검증
- 주요 변경 시 메이저 버전 업

**Module은:**
- 자유롭게 개발
- 독립적으로 배포
- 실패해도 다른 모듈에 영향 없음

**Plugin은:**
- 실험 영역
- 불안정해도 OK
- 사용자가 선택

---

## 3. 모듈 중심 구조

### 설계 철학

**전통적 방식 (❌):**
```
apps/
  ├── api/
  │   └── features/
  │       ├── ledger.ts
  │       └── subscription.ts
  └── web/
      └── pages/
          ├── Ledger.tsx
          └── Subscription.tsx
```

**모듈 중심 (✅):**
```
modules/
  ├── ledger/
  │   ├── frontend/
  │   └── backend/
  └── subscription/
      ├── frontend/
      └── backend/
```

### 장점

1. **명확한 경계**
   - 모듈별로 코드가 모여있음
   - 의존성이 명확함

2. **독립적 개발**
   - 모듈 개발자는 해당 폴더만 수정
   - 다른 코드 건드릴 필요 없음

3. **쉬운 추가/제거**
   - 폴더 하나 추가 = 기능 추가
   - 폴더 하나 삭제 = 기능 제거

4. **커뮤니티 확장**
   - 누구나 모듈 개발 가능
   - GitHub 레포로 공유
   - 마켓플레이스에 등록

## 4. 개인정보 보호 우선

### 핵심 원칙

**사용자가 직접 관리:**
- DB 연결 정보
- AI API Key
- OAuth 인증 정보
- 통합 서비스 API Key

**프로젝트는 절대:**
- ❌ API Key 수집하지 않음
- ❌ 사용 통계 수집하지 않음
- ❌ 개인정보 전송하지 않음
- ❌ 외부 서버와 통신하지 않음 (업데이트 제외)

### 데이터 주권

**모든 데이터는 사용자 소유:**
- Self-hosted 전제
- 사용자의 서버에 저장
- 언제든 백업/이전 가능
- RAW 데이터 투명하게 공개

---

## 5. 최소한의 의존성

### 의존성 관리 철학

**Core:**
- 필수 의존성만 포함
- 검증된 라이브러리만 사용
- 버전 업데이트 신중하게

**Module:**
- 자유롭게 의존성 추가 가능
- 모듈 폴더 내에서 관리

**목표:**
- Core는 가볍게
- 보안 취약점 최소화
- 빠른 설치 시간

---

## 6. 기능 제한 없음

### 완전 무료 원칙

**절대 하지 않는 것:**
- ❌ 프리미엄 기능
- ❌ 유료 모듈
- ❌ 기능 제한
- ❌ 사용자 수 제한
- ❌ 광고

**누구나 동일하게:**
- ✅ 모든 기능 사용 가능
- ✅ 무제한 모듈 설치
- ✅ 무제한 사용자
- ✅ 상업적 이용 가능

### 지속 가능성

**수익 모델 없이 지속:**
- 선택적 후원 (GitHub Sponsors)
- 투명한 재정 운영
- 커뮤니티 기여
- 장기적 비전

---

## 7. 확장성과 유지보수

### 미래를 고려한 설계

**확장 가능:**
- 모듈 시스템으로 무한 확장
- 플러그인으로 실험적 기능
- 통합 서비스 계속 추가

**유지보수 쉽게:**
- 명확한 구조
- 자동화된 테스트
- 문서화
- 버전 관리

### 기술 부채 최소화

**원칙:**
- 단기적 해결보다 장기적 설계
- 복잡도 최소화
- 표준 패턴 사용
- 리팩토링 주기적 진행

## 10. 점진적 개선

### 완벽보다 진행

**MVP 먼저:**
- 기본 기능부터 구현
- 빠르게 출시
- 사용자 피드백 반영

**지속적 개선:**
- 작은 업데이트 자주
- 사용자 요청 반영
- 버그 신속 수정

**장기 비전:**
- 개인 생산성 OS
- 모든 것을 관리하는 플랫폼
- 하지만 천천히, 안정적으로

---

## 원칙 우선순위

1. **개인정보 보호** (절대 타협 불가)
2. **투명성** (모든 것을 공개)
3. **기능 제한 없음** (완전 무료)
4. **안정성** (Core는 절대 깨지면 안 됨)
5. **확장성** (모듈로 무한 확장)
6. **커뮤니티** (함께 만듦)

이 원칙들이 충돌할 때는 위 순서대로 우선순위를 둡니다.
